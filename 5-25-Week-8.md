That aside, let's look at where progress was made!

Last we left off, I was looking into why `femto` was panicking during its core loop, where it was attempting to run a function `print_screen()`, which runs `termion`'s `terminal_size()` function, which calls `ioctl()` with the register `TIOCGWINSZ` or **T**erminal **I**nput / **O**utput **C**onsole **G**et **WIN**dow **S**i**Z**e. Luckily, there is already a branch in `mlibc`'s `sysdeps.cpp` that handles `ioctl` when it's called with that register, so no issues on the `mlibc` level it turns out (*phew*)

Where things actually went wrong is in the way Twizzler handles `get_config()` when it gets Twizzler's version of `TIOCGWINSZ`, `IO_REGISTER_WINSIZE`, in `src/rt/reference/src/runtime/file/kinds/pty.rs`. 

There wasn't any branch to handle a `IO_REGISTER_WINSIZE` call yet, so it was funnelled into the catch-all that returns `Err(TwzError::INVALID_ARGUMENT)`, thus causing a panic.

Luckily, a lot of the logic that getting and setting the terminal's window size needs is already somewhat implemented for getting and setting `termios`, a `libc` API for terminal communications. However, window size requires its own `libc` struct, `winsize` and its respective functions. So I added a `winsize` struct to the `PtyBase` struct that serves as the base for the PTY as the name would suggest. Following `termios`'s implementation I added a `winsize_gen` and functions for seqlock functionality as `winsize` will be a shared object between multiple apps just as `termios` is. After that, it was just creating getter and setter functions for `winsize` based on the ones that exist for `termios` inside the `PtyClientHandle` and `PtyServerHandle`.

One important disctinction between `termios` and `winsize` is that, unlike `termios`, `winsize` does not need to be cached locally between the client and server as it doesn't touch input or output like `termios` does, so `winsize` omits an functions relating to caching and invalidating outdated caches.

Boom! Creating handling for the `IO_REGISTER_WINSIZE` branch and setting `winsize` to be some default terminal size (i.e. 80x24) and `femto` was up and running! Through my process I also ended up adding functionality for `TIOCSWINSZ`, which is the "setter" version of `TIOCGWINSZ`, although nothing in `femto` or `termion` use this.

There is, however, one small problem with `femto` that I attempted to solve through multiple venues, which is actually getting the size of the terminal. Setting the default terminal size to 80x24 is fine, but its not how `femto`, or really any text editor, is made to be. This proved to be a whole lot more of a challenging task.

Due to the amount of layers of abstraction between the User's terminal and Twizzler's PTY, some features get lost in translation. The most important of these features is the POSIX signal, `SIGWINCH`, which stands for **SIG**nal **WIN**dow **CH**ange and is sent out everytime the user changes their terminal's window size. Its also brought my attention to the word "winch" and how I haven't heard that word in a while.

The way Twizzler currently initializes its PTY and its underlying `KernelConsole` is through a QEMU-emulated UART connection (16550 UART at I/O port `0x3f8` accessed byte by byte for x86 and PL011 UART via MMIO for ARM). Due to this old-school UART emulation, signals like `SIGWINCH` that QEMU try to send are simply out-of-band for the UART and are dropped.

Not wanting to give up, I still implemented what I think `SIGWINCH` would do, could it be somehow passed through to the PTY. So the functionality is there... it just needs to be called upon...

And it is! (Somewhat)

Due to this UART restriction, the only reasonable way to find out the User's terminal size without overhauling Twizzler's console drivers, is to query the User's terminal, whatever it may be, for its size in characters with the escape code `\x1b[18t`. We can then parse the response for the width and height, add it to the `winsize` struct's properties, and then remove the output before it's sent to `stdout` and thus the user.

I did this by implementing a state machine inside the init script found at `src/bin/init/src/main.rs` that looks for `\x1b` in the raw input buffer and enters "intercept" mode to parse the escape code and remove it from the buffer if necessary.

When the init thread does find a request for terminal size and sends a call for `twizzler_rt_abi::bindings::twz_rt_fd_set_config()` with the newly updated `winsize` struct, this also "artificially" sets off the `SIGWINCH` signal for the PTY! Which currently is only picked up by `src/bin/shell/src/main.rs` (and ignored) and `femto` (where it doesn't do much either). Despite this somewhat anti-climactic ending, I find it cool that applications at the PTY level are even able to recognize that `SIGWINCH` is being sent out and that it's even being sent out in the first place.

Now `femto` properly scales to the user's terminal size *on start-up*, but if the user makes their terminal smaller or bigger during their use of Twizzler, then they're out of luck. This is where a real `SIGWINCH` signal from the user would be the solution, but since we don't really have a way of telling exactly when a user changes their terminal size, I decided to just have a thread on init that polls every 3 seconds for the terminal size and sets it accordingly. Not too frequently, but not too infrequently. And, hey, at least changing the terminal size after start-up won't destroy `femto` now! (at least after about ~2-3 seconds)

### Troubles

Obviously, I ran into a bunch of trouble trying to figure out how to get dynamic terminal sizing within `femto` with the now infamous `SIGWINCH` signal. I also had a bit of a woopsie with `femto` having an overflow error when the terminal's `winsize` is set to a width and height of 0x0. It tries to subtract 2 from 0 in a for loop to print empty lines and suddenly ends up with an 18 billion line terminal...
