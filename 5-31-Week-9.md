I also worked on a little toolchain issue I mentioned last week regarding the `--step libc` option on the toolchain boostrap command. Since I don't think Daniel is pulling the toolchain from GitHub and then using the `--step libc` flag to change a part of the pulled toolchain, it makes sense that this would fly under his radar. But, yeah, essentially when you pull the toolchain from GitHub it embeds a bunch of hardcoded filepaths into it that xtask tries to use and fails to do since they only exist for the GitHub Actions Runner. So I made some changes to the toolchain that use the local filepaths instead of those. There was also an issue with the python environment not being correctly implemented to use meson. I then wrote a Pull Request on this and it's been merged!

I've started working on another toolchain change that I think would be particularly helpful for development. There have been times where I've made a small change to `mlibc`, which causes me to have to have a new toolchain, but I won't have the base toolchain that I could easily run `--step libc` on, forcing me to either do some github submodule magic or bootstrap the whole toolchain anew. So I've had a vision to make the toolchain more flexible / intelligent when running the `bootstrap` command and be able to suggest downloading a previous toolchain release from GitHub (if it exists) and then running `--step libc` on it to save on time and resources. Currently the toolchain is hardcoded to just look for the most recent GitHub release, with no support for development that may be behind on changes.

Relevant files have been:
`tools/xtask/src/toolchain/utils.rs`
`tools/xtask/src/toolchain/mod.rs`
